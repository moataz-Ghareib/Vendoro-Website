## 🏗️ معايير المعمارية الصارمة (Strict Architecture Guidelines)

لضمان جودة الكود وأداء النظام (Enterprise-Grade)، نلتزم بالمعايير التالية بشكل قاطع:
1. **N+1 Queries:** ممنوع نهائياً استخدام استعلامات الداتا بيز داخل الـ Loops. يتم استخدام (Eager Loading) عبر `with()` أو الاستعلامات المجمعة `whereIn` لتجهيز البيانات مسبقاً.
2. **Magic Strings:** ممنوع استخدام النصوص الثابتة للحالات (مثل `'sale_out'`). نستخدم دائماً **Enums** مدعومة بقوة الـ PHP 8.1 (مثل `StockMovementType::SALE_OUT`).
3. **Mass Assignment:** يُمنع تماماً استخدام `$guarded = ['id']`. جميع الجداول يجب أن تستخدم مصفوفة `$fillable` دقيقة وصريحة لحماية البيانات الحساسة (خصوصاً جداول الـ Stock و Financials).
4. **Filament/Closures Memory Leaks:** تجنب حَل الحاويات (Container Resolution) بشكل متكرر داخل الـ `closures` المتعددة للفورم (مثل استدعاء `app(Service::class)` داخل `visible()` أو `required()`). يُفضل استخدام Session Helpers الخفيفة مثل `auth()->user()`.
5. **No env() outside config:** قراءة الإعدادات تكون من الـ `config()` فقط، لضمان دعم `config:cache` في بيئة الإنتاج.
6. **Fat Controllers:** تُكتب بيئة العمليات الحساسة (كالتسعير وسحب المخزون) في **Services** منعزلة ويتم تغليفها بـ `DB::transaction()` لضمان تكامل البيانات.

تم تطبيق هذه المعايير (Refactoring ✅) على `Recipe` و `StockMovement` و `InventoryService` و `ProductForm` بنجاح!

### 🛡️ المعمارية الحالية (Enterprise Architecture - يونيو 2026 ✅)
يتمتع النظام حالياً ببنية تحتية بمستوى SaaS السحابي المتقدم، قادرة على تحمل ملايين الحركات بكفاءة:
1. **أمان البيانات (Data Integrity):**
   - استخدام الـ `restrictOnDelete` في جميع الجداول الأساسية لمنع الحذف العرضي لبيانات الشركات وحماية السجلات المالية.
   - تفعيل نظام الاسترجاع والأرشفة `SoftDeletes` بشكل شامل لكل الكيانات (`Company`, `Branch`, `Product`, `Recipe`, `StockMovement`, `InventoryBalance`).
2. **الأداء اللاتزامني للواجهة (Asynchronous UI):**
   - الاعتماد على الـ `Queue Jobs` (مثل `UpdateRecipesCostJob`) في معالجة العمليات الحسابية المتسلسلة لضمان استجابة الواجهة فوراً للعميل.
   - استخدام أحداث الـ `Model Events` السليمة لتوجيه التحديثات بدقة بين مكونات النظام.
3. **أداء المخزون اللحظي O(1) (Inventory Engine):**
   - يمتلك النظام جدول أرصدة مخصص (`inventory_balances`) مزود بفهارس مركبة `Unique Compound Index` يضمن جلب الأرصدة بسرعة فائقة `O(1)`، وهو جاهز للربط مع شاشات الـ POS والـ APIs الخارجية بأداء لا يتعدى 50ms.
4. **العزل المتعدد للشركات (Native Multi-Tenancy):**
   - النظام مبني على معيار `Filament Native Tenancy` العالمي لتوفير عزل أمني مطلق (100%) لبيانات كل شركة على حدة حتى في الخلفية (Background Jobs) ولحظات التحديث الخفية للواجهة (Livewire AJAX).
   - توجيه الروابط يعتمد على الأسماء المختصرة (`slug`) بدلاً من الأرقام لتعزيز الهوية البصرية للشركات (White-labeling).
   - علاقة مرنة (Many-to-Many) بين المستخدمين والشركات عبر جدول `company_user`، تتيح للمستثمر إدارة فروع وشركات مختلفة (مخبز، محل موبايلات) من نفس الحساب.

## 4. مقارنة المنطق التجاري بين القطاعين

> هذا القسم يوضح لماذا لا يمكن دمجهما في Sprint واحد.

### 4.1 دورة حياة المنتج

```
المخبز:
مواد خام (دقيق + سكر + بيض)
    ↓ [Production Batch]
منتج نهائي (كيكة 50 قطعة)
    ↓ [Sale]
فاتورة بيع → خصم المخزون → تسجيل الإيراد
    ↓ [Expiry]
هدر إذا لم تُباع قبل انتهاء الصلاحية

الموبايل:
شراء جهاز (IMEI: 354821...)
    ↓ [Purchase In]
جهاز في المخزون (حالة: available)
    ↓ [مساران: بيع أو عُهدة]
المسار أ: بيع (Sale) → الجهاز (sold) → ضمان يبدأ
المسار ب: عُهدة (Memo) → الجهاز (pending_out) → إما يُباع لاحقاً أو يُسترجع
    ↓ [طرق الدفع]
نقدي / أقساط مجدولة / ذمة مالية (دفعات حرة وتسجيل المتبقي في كشف الحساب)
```

### 4.2 شاشة POS

| المعيار | المخبز | الموبايل |
|---------|--------|----------|
| طريقة الإضافة | أزرار كبيرة بصرية (Grid) | بحث بالـ IMEI أو الباركود |
| سرعة البيع | ثوانٍ — عملاء كثيرون | دقائق — كل جهاز له تفاصيل |
| الكميات | متعددة — 5 كيكات | دائماً 1 لكل IMEI |
| الدفع | كاش أو كارت فقط | كاش، كارت، أقساط مجدولة، ذمم مالية (آجل) |
| الضغط | عالٍ جداً — ساعات الذروة | متوسط — بيع واحد كل 15 دقيقة |
| الطابعة | إيصال بسيط | فاتورة تفصيلية + ضمان |

### 4.3 تكلفة المنتج

```
المخبز:
تكلفة الكيكة = (200جم دقيق × سعر الكجم)
              + (100جم سكر × سعر الكجم)
              + (3 بيضة × سعر البيضة)
              + تكلفة الطاقة والعمالة (نسبة ثابتة)
← تحسب تلقائياً من الوصفة

الموبايل:
تكلفة الجهاز = سعر الشراء الفعلي لهذا الـ IMEI تحديداً
← لكل جهاز تكلفة مختلفة حتى لو نفس الموديل
```

### 4.4 التقارير الحيوية

| التقرير | المخبز | الموبايل |
|---------|--------|----------|
| الأهم يومياً | الهدر + الإنتاج vs المبيعات | المخزون بالـ IMEI |
| الأهم شهرياً | تكلفة المواد الخام | الأقساط المستحقة |
| تنبيه حرج | انتهاء الصلاحية قريباً | قسط متأخر |
| هامش الربح | صعب — يعتمد على الهدر | سهل — سعر بيع - سعر شراء |

## ملاحظات ختامية للمطور

### قرارات لا تتراجع عنها (Architectural Standards)
- **Cart في Redis دائماً** — لا session، لا database مباشرة في POS (لمنع تداخل السلات).
- **POS UI معزول تماماً عن Filament** — layout وتصميم مستقل، لكن مع **نفس الحماية الأمنية والجلسة** (لا استثناء لـ CSRF).
- **Stock Movement فقط** — لا تعديل مباشر على أرقام المخزون أبداً
- **company_id في كل query** — حتى لا تختلط بيانات الشركات
- **قاعدة PostgreSQL JSON:** يُمنع تماماً استخدام `json()` في الـ Migrations، واستخدم دائماً وبشكل صارم `jsonb()` لضمان السرعة الخرافية في البحث وفهرسة `GIN`.
- **ترجمة Filament (Spatie Translatable):** يُمنع استخدام طريقة الـ Dot Notation (مثل `name.ar` و `name.en`) في الحقول المترجمة لتجنب ازدحام الواجهة. استخدم دائماً اسم الحقل الأساسي (`name`) مع الـ Translatable plugin (Locale Switcher).
  - *استثناء وحيد لهذه القاعدة:* إذا كان مدير النظام يحتاج "إجبارياً" لرؤية النصين (العربي والإنجليزي) جنباً إلى جنب في نفس اللحظة للمقارنة أو الترجمة الفورية الدقيقة المعقدة، فيمكن وقتها استخدام حقلين منفصلين. عدا ذلك، استخدم الـ Switcher.

### ترتيب أولويات Bug Fixing
1. أي bug يتعلق بأمان البيانات (تسريب بين شركات) → أولوية قصوى
2. أي bug في حسابات المخزون → أولوية عالية
3. أي bug في شاشة POS → أولوية عالية (يوقف البيع)
4. أي bug في التقارير → أولوية متوسطة

### للإطلاق الأول — أبسط من أن تعتقد
لا تحتاج كل شيء مثالياً. تحتاج فقط:
- شاشة POS تعمل بثقة تحت الضغط
- المخزون يخصم بدقة
- التقرير اليومي صحيح

كل شيء آخر يمكن تحسينه بعد الإطلاق.

## 🚀 المرحلة الرابعة — التطوير المتقدم والاحترافي (The Enterprise Phase)
هذه الخريطة مصممة لسد الفجوات المعمارية المتبقية وتحويل النظام إلى نظام ERP متكامل لا يُقهر.

### 🛠 Sprint 16: الدرع المحاسبي (نظام المرتجعات والتسويات)
الهدف: ضمان عدم وجود أي خطأ تشغيلي بدون معالجة محاسبية عكسية صحيحة.

- [x] **إنشاء موديول مرتجعات المبيعات (Sale Returns):**
  - تصميم `SaleReturn` و `SaleReturnItem` Models.
  - برمجة `SaleReturnObserver` لإعادة الكميات للمخزون (Stock Movement).
  - إضافة دالة في `AccountingService` لتسجيل القيد العكسي (تخفيض الإيراد، استرداد COGS، استرداد المخزون، وتخفيض النقدية أو رصيد العميل).
- [x] **إنشاء موديول مرتجعات المشتريات (Purchase Returns):**
  - تصميم النماذج والربط مع `PurchaseInvoice`.
  - برمجة القيد العكسي (تخفيض المخزون، تخفيض ذمم الموردين).
- [x] **تحديث شاشات Filament:** إضافة أزرار "إرجاع" (Refund) في تفاصيل الفواتير القديمة.

### 🛠 Sprint 17: ذكاء المخزون (Inventory Valuation Engine)
الهدف: دقة حساب الأرباح والخسائر عند تقلب أسعار الشراء.

- [x] **خوارزمية متوسط التكلفة (Moving Average Cost):**
  - إنشاء `InventoryValuationService`.
  - عند كل عملية شراء جديدة (`PurchaseInvoice`)، يقوم النظام بإعادة حساب حقل `cost_price` للمنتج بناءً على الكمية الموجودة والتكلفة الجديدة.
  - تعديل `SaleObserver` ليسحب الـ COGS بناءً على متوسط التكلفة اللحظي وقت البيع، وليس سعر التكلفة الثابت.
- [x] **التسويات الجردية (Stock Adjustments):**
  - موديول لتسوية عجز أو زيادة المخزون بعيداً عن البيع والشراء (مع قيد يومية يرمي على حساب خسائر تلف/عجز مخزون).
- [ ] **ملفات تيست قوية وقاسية (Strict & Hard Tests):**
  - كتابة اختبارات معقدة لضمان دقة دالة حساب متوسط التكلفة وحركات التسوية المحاسبية.

### 🛠 Sprint 18: واجهة الكاشير المتقدمة (POS Revamp)
الهدف: القضاء على بطء الإدخال واستبدالها بواجهة كاشير سريعة حية.

  - تصميم واجهة صفحة واحدة (SPA) تحتوي على المنتجات (Grid)، سلة المشتريات (Cart)، وحاسبة الدفع.
- [x] **دعم قارئ الباركود (Barcode Scanner Integration):**
  - برمجة نظام يستمع لإدخالات الباركود ويضيف المنتج للسلة فوراً.
- [x] **التحكم بالورديات المحكم:**
  - الكاشير لا يمكنه فتح شاشة הـ POS إلا إذا كان هناك `Shift` مفتوح باسمه.

### 🛠 Sprint 19: أتمتة التصنيع والمخابز (Manufacturing & Recipes)
